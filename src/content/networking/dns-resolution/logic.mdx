## What DNS resolution solves

DNS translates human-readable domain names (like example.com) into machine-routable IP addresses. Without DNS, users would need to memorize numeric IP addresses for every website.

## Steps

1. The browser asks one **recursive resolver** for `A example.com`.
2. The resolver checks its cache. A cache hit ends the lookup immediately.
3. On a miss, the resolver asks a **root** server where `.com` is served.
4. The root returns a **referral** to `.com` nameservers — not the final IP.
5. The resolver asks `.com`, which refers it to the **authoritative** nameserver for `example.com`.
6. The authoritative server returns the actual A record and its **TTL**.
7. The resolver caches that record, then returns the IP to the browser.

> A referral says *who to ask next*. An answer contains the record the client asked for. The simulation keeps those two response types visibly separate.

## Pseudocode

~~~c
typedef struct {
    char ip[16];
    int ttl;
    bool found;
} DNSRecord;

DNSRecord resolveA(Resolver *resolver, char *qname) {
    DNSRecord cached = cacheGet(resolver->cache, qname);
    if (cached.found) return cached;

    DNSReply root = ask(ROOT_SERVER, qname, "A");
    Server tld = root.referral;

    DNSReply com = ask(tld, qname, "A");
    Server authority = com.referral;

    DNSReply reply = ask(authority, qname, "A");
    cachePut(resolver->cache, qname, reply.ip, reply.ttl);
    return reply;
}
~~~

## Common pitfalls

- Cache poisoning — an attacker injects a fake DNS record into a resolver's cache.
- Assuming DNS resolution always succeeds — networks may have blocked or misconfigured resolvers.
- Ignoring TTL values — stale cached records can cause connectivity issues after IP changes.
- Treating a referral as an answer — root and TLD servers normally direct the resolver onward instead of returning the site's A record.
- DNS over TCP vs UDP — large responses require TCP fallback when the truncation bit is set.
