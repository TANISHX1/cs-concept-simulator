## What async/await solves

I/O-bound programs spend most of their time waiting — for network responses, disk reads, or timers. Blocking the thread during each wait wastes CPU. Async/await lets a function pause without blocking the thread, so other work can proceed, and automatically resume when the awaited operation completes.

## How it works

The runtime maintains:
- **Call stack**: the currently executing frames.
- **Microtask queue**: promise callbacks and resumed async frames (higher priority).
- **Macrotask queue**: timers, I/O callbacks (lower priority).

The event loop repeatedly: run all microtasks, then pick one macrotask, run it to completion, then repeat.

## Pseudocode

~~~javascript
async function fetchData(url) {
    const response = await fetch(url);   // suspend here
    const data     = await response.json();
    return data;
}

function main() {
    console.log("start");
    const promise = fetchData("/api/data");
    console.log("after async call");      // executes before fetchData resumes
    promise.then(data => console.log(data));
    console.log("end");
}

main();
// Output order:
//   "start"
//   "after async call"
//   "end"
//   (later) data from fetch
~~~

## Common pitfalls

- **Forgotten `await`**: calling an async function without `await` returns a promise; errors become unhandled rejections.
- **Blocking the event loop**: a synchronous CPU-heavy operation inside an async function stalls all tasks.
- **Race conditions**: shared mutable state between concurrent async functions without synchronization.
- **Promise chain vs await**: mixing `.then()` and `await` in the same function can lead to confusing execution order.
- **Unhandled rejections**: an awaited promise that rejects throws; if not caught, the error is swallowed.
