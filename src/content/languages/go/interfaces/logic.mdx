## What Go interfaces solve

Object-oriented languages typically require explicit `implements` declarations, coupling types to interfaces at compile time. Go's structural typing means a type satisfies an interface automatically if it has the required methods — enabling polymorphism without class hierarchies or import dependencies.

## How it works

An interface `I` with method set `M` is a protocol. Any concrete type `T` whose method set is a superset of `M` can be assigned to a variable of type `I`. The compiler checks this at compile time — no runtime type checking is needed for assignment.

## Pseudocode

~~~go
// Interface definition
type Stringer interface {
    String() string
}

// Concrete types — no "implements" keyword needed
type Book struct {
    Title string
}

func (b Book) String() string {
    return "Book: " + b.Title
}

type Person struct {
    Name string
}

func (p Person) String() string {
    return "Person: " + p.Name
}

// Interface value — holds dynamic type and value
var s Stringer = Book{Title: "Go in Action"}
fmt.Println(s.String()) // dynamic dispatch → Book.String()

s = Person{Name: "Alice"}
fmt.Println(s.String()) // dynamic dispatch → Person.String()

// Type assertion
book, ok := s.(Book)   // extract concrete type
if ok {
    fmt.Println(book.Title)
}

// Type switch
switch v := s.(type) {
case Book:
    fmt.Println("book:", v.Title)
case Person:
    fmt.Println("person:", v.Name)
default:
    fmt.Println("unknown")
}

// Empty interface — can hold any value
var any interface{}
any = 42
any = "hello"
any = Book{}
~~~

## Common pitfalls

- **Nil interface vs nil pointer**: a nil pointer of type `*Book` assigned to an `io.Writer` interface produces an interface with non-nil type info but nil data — checking `if writer == nil` returns false, causing surprising nil dereferences.
- **Large interface definitions**: interfaces with many methods are hard to satisfy and violate the Go philosophy of small interfaces. Prefer many small interfaces (1-3 methods) over one large one.
- **Interface pollution**: defining interfaces before they are needed, or for every struct, adds indirection without benefit. Let consumers define the interfaces they need (implicit satisfaction enables this).
- **Value receiver vs pointer receiver**: a type with pointer-receiver methods may not satisfy an interface that expects value-receiver methods (though the converse works — the compiler auto-generates pointer-receiver methods for value receivers).
