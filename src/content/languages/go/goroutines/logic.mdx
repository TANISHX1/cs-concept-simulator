## What goroutines solve

Threads in most languages are heavyweight (MB-sized stacks, slow creation/context switching). Goroutines are user-space threads multiplexed onto OS threads — they start with a tiny stack (~4 KB) and are scheduled by Go's runtime, making it practical to launch hundreds of thousands of concurrent tasks.

## How it works

The `go` keyword launches a new goroutine — it is not a function call but a concurrent execution unit scheduled by the Go runtime's M:N scheduler (M OS threads, N goroutines). Channels are typed conduits that connect goroutines.

## Pseudocode

~~~go
// Unbuffered channel — synchronous
ch := make(chan int)

go func() {
    result := doWork()
    ch <- result // blocks until main receives
}()

value := <-ch // blocks until worker sends
fmt.Println(value)

// Buffered channel — asynchronous up to capacity
bufCh := make(chan string, 3)
bufCh <- "a" // does not block
bufCh <- "b" // does not block
bufCh <- "c" // does not block
// bufCh <- "d" would block (buffer full)

// Select — wait on multiple channel operations
select {
case v := <-ch:
    fmt.Println("received:", v)
case <-time.After(1 * time.Second):
    fmt.Println("timeout")
}
~~~

## Common pitfalls

- **Deadlock**: all goroutines are blocked waiting on each other. The runtime detects this and crashes with a stack trace, but only if the program is truly stuck.
- **Goroutine leak**: a goroutine blocked on sending to a channel that nobody reads will never be garbage collected. Always ensure receivers exist.
- **Closing channels**: sending on a closed channel panics. Receiving from a closed channel returns the zero value immediately (with `ok = false`).
- **Nil channel**: reading from or writing to a nil channel blocks forever — useful in select statements to disable a case.
- **Mutex vs channel**: not everything needs a channel. Use `sync.Mutex` for protecting simple shared state; use channels for signaling and data flow.
