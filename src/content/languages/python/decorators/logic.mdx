## What decorators solve

Cross-cutting concerns — logging, timing, access control, caching — must often be applied to many functions. Copy-pasting the same boilerplate into each function violates DRY and obscures the core logic. Decorators let you attach pre/post behavior declaratively.

## How it works

The `@decorator` syntax desugars to a simple assignment. When multiple decorators stack, they apply bottom-up: the closest decorator to the function definition is applied first.

## Pseudocode

~~~python
# Decorator definition
def log(func):
    def wrapper(*args, **kwargs):
        print(f"calling {func.__name__}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} returned {result}")
        return result
    return wrapper

# Usage with sugar
@log
def greet(name):
    return f"Hello, {name}"

# Desugared equivalent
def greet(name):
    return f"Hello, {name}"
greet = log(greet)

# Stacked decorators
@timer
@log
def fetch_data():
    ...
# Equivalent to: fetch_data = timer(log(fetch_data))
~~~

## Using functools.wraps

~~~python
from functools import wraps

def log(func):
    @wraps(func)            # preserves __name__, __doc__, etc.
    def wrapper(*args, **kwargs):
        ...
        return func(*args, **kwargs)
    return wrapper
~~~

## Common pitfalls

- **Losing metadata**: without `@wraps`, the wrapper replaces `__name__` and `__doc__` of the original function, breaking introspection.
- **Decorator argument confusion**: a decorator that takes arguments (`@deco(args)`) needs an extra nesting layer — it must return a decorator.
- **Stacking order**: decorators apply bottom-up but execute top-down when the function is called — many beginners expect the reverse.
- **Mutating global state in decorators**: side effects in decorators can cause surprising behavior during import time.
