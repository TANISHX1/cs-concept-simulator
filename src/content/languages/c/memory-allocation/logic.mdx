## What manual memory management solves

C requires the programmer to explicitly allocate and free heap memory. This gives full control over when memory is reserved and released, matching the program's actual lifetime needs — essential for systems programming, embedded devices, and performance-critical code where garbage collection overhead is unacceptable.

## Core idea

Manual memory management works through four key functions that operate on the heap region of a process's address space. The programmer must track every allocation and ensure it is freed exactly once.

## Key functions

| Function | Signature | Behaviour |
|----------|-----------|-----------|
| `malloc` | `void *malloc(size_t size)` | Allocates `size` bytes; contents uninitialized. |
| `calloc` | `void *calloc(size_t num, size_t size)` | Allocates `num * size` bytes; zero-initializes. |
| `realloc` | `void *realloc(void *ptr, size_t new_size)` | Resizes a previous allocation; may move data. |
| `free` | `void free(void *ptr)` | Releases memory back to the heap. |

## Pseudocode

~~~c
#include <stdlib.h>

int *p = malloc(4 * sizeof(int));   // allocate 16 bytes on heap
if (p == NULL) {
    // handle allocation failure
    return 1;
}

p[0] = 10;
p[1] = 20;

int *q = malloc(8 * sizeof(int));

free(p);       // release p's memory
p = NULL;      // nullify to prevent dangling access

int *r = calloc(5, sizeof(int));     // zero-initialized
r = realloc(r, 10 * sizeof(int));    // grow to 10 ints

free(q);
free(r);
~~~

## Common pitfalls

- **Dangling pointer**: using a pointer after the memory it points to has been freed.
- **Double free**: calling `free` on the same pointer twice (undefined behaviour).
- **Memory leak**: forgetting to call `free`; memory remains allocated until process exits.
- **Use-after-free**: reading or writing through a freed pointer (often exploitable).
- **Buffer overflow**: writing past the end of an allocated region (heap corruption).
- **Not checking malloc return**: `malloc` can return `NULL` on failure.
