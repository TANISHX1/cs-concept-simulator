1. PROJECT OVERVIEW
================================================================================
Name: CS Concept Simulator (working title — agent may propose alternatives)

What it is: A web application for learning computer science concepts through
interactive visual simulations. Organized into top-level sections (Algorithms,
Operating Systems, Networking, System Programming, Languages, etc.), each
containing a hierarchy of concepts and sub-concepts, similar to a file/directory
tree. Each concept page contains: an interactive simulation, a logic/explanation
write-up, and a curated references list. A persistent AI chat side-panel is
available throughout the working area to help the user understand the concept
they are currently viewing.

Core requirements:
- NO login / NO authentication anywhere. App opens directly into content.
- Landing page with a 3D animated hero, then a direct entry into the working
  "Workspace" area (IDE-like 3-pane layout).
- Must be trivially extensible: adding a brand-new concept/simulation should
  require ONLY adding new content files — no changes to routing code, sidebar
  code, or app shell code.
- Visual identity: sharp, professional, "native OS app" feel — NOT a generic
  marketing website. Reference aesthetic: Vercel Dashboard / Linear / VS Code.
- Two themes: exact Vercel-style dark theme, and a warm "buttery" light theme.
  Full color system detailed in Section 6.

2. TECH STACK [LOCKED]
================================================================================
- Build tool / framework: Vite + React (SPA). Do NOT use Next.js. Do NOT mix
  Vite and Next.js — they are mutually exclusive framework choices.
- Language: TypeScript throughout (strict mode on).
- Routing: React Router v6 (client-side only).
- Styling: Tailwind CSS + shadcn/ui (Radix primitives underneath).
- 3D graphics (landing page only): @react-three/fiber + @react-three/drei
  (React wrappers over Three.js — do not use raw three.js imperatively).
- Simulation animation library (default/shared): Framer Motion, for consistent
  transition language across all simulations. Individual simulations may layer
  SVG, Canvas (via Konva.js if needed), or D3.js underneath as required by the
  concept type (see Section 10 for the mapping).
- Content authoring format: MDX (for logic write-ups that may embed live
  React components inline).
- AI chat backend: ONE serverless function (see Section 9). Provider: Google
  Gemini API. The API key lives ONLY in the serverless function's environment
  variables — it must NEVER be shipped to or callable from client-side code.
- Icons: Lucide React.
- Fonts: Inter or Geist for UI text; JetBrains Mono or IBM Plex Mono for code/
  pseudocode blocks.
- Command palette: cmdk library (pairs natively with shadcn).
- Deployment target: Vercel or Netlify (both support static SPA hosting +
  multiple serverless functions in the same deploy — no separate backend
  service required). See Section 2.1 for the full list of functions.
- Concept-request backend: GitHub Issues API, called from a dedicated
  serverless function using a repo-scoped token. Full spec in Section 17.
- Code execution trace tooling (existing concepts only, offline/dev-time
  only — NOT a production service): OnlinePythonTutor
  (github.com/pgbovine/OnlinePythonTutor, GPLv3), used locally by content
  authors to generate precomputed trace.json files. Full spec in Section 18.
- "Simulate the Logic" (Beta/Phase 2) backend: Cloudflare Workers, proxying
  to an LLM with a constrained JSON output schema. Full spec in Section 19.

2.1 Serverless functions overview (all live under /api, all deploy together)
  /api/chat.ts               — AI chat panel proxy (Gemini). See Section 9.
  /api/request-concept.ts    — Concept request form -> GitHub Issues. Section 17.
  /api/simulate-logic.ts     — Beta "Simulate the Logic" page proxy (may be
                                implemented as a Cloudflare Worker instead of
                                a Vercel/Netlify function — see Section 19 for
                                the tradeoff). Section 19.
  All three are independent, stateless, single-purpose functions. None of
  them share code beyond the common fetch/error-handling helpers in
  /src/lib/api.ts (client side) — keep them decoupled so any one of them
  can be modified or redeployed without touching the others.

2.2 Team development environment
  Primary dev OS for this team: Fedora Linux (all current contributors).
  This matters specifically for Section 18 (offline code-trace authoring),
  which needs a local Docker-compatible runtime:
  - Fedora ships Podman by default, not Docker. Podman is Docker-CLI
    compatible for this project's purposes (`alias docker=podman` works for
    the commands in Section 18, or use `podman-docker` package which
    provides a real `docker` shim: `sudo dnf install podman-docker`).
  - If a contributor prefers actual Docker Engine on Fedora instead of
    Podman, it is not in Fedora's default repos — install via Docker's
    official Fedora repo, not `dnf install docker` (that pulls an unrelated
    package called `docker` from Fedora's repos in some releases). Podman is
    the simpler, pre-installed path and is what this spec assumes by default
    in Section 18's commands.
  - Nothing else in the stack (Vite, Node, the serverless functions) is
    OS-specific — this note only applies to the offline trace-generation
    tooling.

Explicitly OUT of scope / do not add:
- No Redux / Zustand / MobX — state needs are minimal (see Section 11).
- No database. No user accounts. No persistent server-side storage.
- No SSR/SSG — this is a pure client-rendered SPA.

4. CONTENT PLUGIN ARCHITECTURE (CORE MECHANISM) [LOCKED]
================================================================================
This is the most important architectural pattern in the app. It is what
guarantees "adding a new concept = only add files, no code changes."

4.1 Folder convention
Every concept (and sub-concept) is a folder under /src/content/<section>/
containing these files:
  - meta.json          (required)
  - Simulation.tsx      (required)
  - logic.mdx           (required)
  - references.json     (required, may be an empty array initially)
  - trace.json          (OPTIONAL — enables the synced simulation + code
                          step-through view described in Section 18. Its
                          mere presence in a concept folder is what triggers
                          the split view in the UI; concepts without it
                          behave exactly as they did before this feature
                          existed — fully backward compatible.)

Nesting is allowed arbitrarily deep to represent hierarchy, e.g.:
  /content/algorithms/graph/shortest-path/dijkstra/...
The registry builder must support arbitrary depth, not just 2 levels.

4.2 meta.json schema
{
  "id": "shortest-path",                 // must match folder name, kebab-case
  "title": "Shortest Path Algorithms",
  "section": "algorithms",               // top-level section key
  "parentPath": "algorithms/graph",      // full path minus this folder, for
                                          // breadcrumb + sidebar nesting
  "order": 3,                            // sort order within its parent
  "difficulty": "intermediate",          // "beginner" | "intermediate" | "advanced"
  "tags": ["graph", "greedy", "dijkstra", "bellman-ford"],
  "prerequisites": ["algorithms/graph/bfs-dfs"],  // array of concept ids/paths
  "summary": "One or two sentence description shown in sidebar hover / search results",
  "accentSection": "algorithms"          // used to pick the correct accent
                                          // color token (see Section 6.4)
}

4.3 Simulation.tsx contract
- Must be a default-exported React component with NO required props (it may
  accept optional props for things like initial state, but must render fully
  standalone with defaults).
- Must be self-contained: all state, animation logic, and controls (play/
  pause/step/reset/speed slider) live inside this file or its own local
  sub-components in the same folder if it grows large (e.g. create a
  /shortest-path/components/ subfolder for internal pieces — this is allowed
  and does not break the plugin contract, since the loader only ever imports
  Simulation.tsx as the entry point).
- Must use Framer Motion for transitions to keep visual language consistent
  app-wide, and must consume the theme's CSS variables for colors (never
  hardcode hex values inside a simulation file).
- Must implement standard simulation controls as a shared UI convention:
  Play / Pause / Step Forward / Step Back / Reset / Speed control. Build a
  reusable <SimulationControls /> component in /components/ui and have every
  Simulation.tsx use it, so all simulations feel consistent.
- OPTIONAL extension for Section 18: Simulation.tsx MAY accept an
  `externalStep?: number` prop. When provided (this only happens when a
  sibling trace.json exists and ConceptWorkbench.tsx is driving the view),
  the component must render exactly that step instead of managing its own
  internal step counter, and ConceptWorkbench takes over rendering
  <SimulationControls /> itself. When the prop is absent, Simulation.tsx
  behaves exactly as originally specified — this is purely additive.

4.4 logic.mdx contract
- Contains the written explanation: problem definition, step-by-step logic,
  pseudocode (in fenced code blocks using the mono font), time/space
  complexity, common pitfalls.
- Since it's MDX, it MAY embed small inline React components (e.g. a mini
  complexity comparison table component) — but the main interactive
  simulation always lives in Simulation.tsx, not inline in the MDX.

4.5 references.json schema
[
  {
    "title": "Dijkstra's Algorithm — CLRS Chapter 24",
    "type": "book",                 // "paper" | "video" | "doc" | "tool" | "book" | "article"
    "source": "Introduction to Algorithms (CLRS)",
    "url": "https://...",
    "verified": true                // manually curated/verified flag, shown as a badge
  }
]

4.6 registry.ts — auto-build logic
Implemented in /src/lib/contentLoader.ts using Vite's import.meta.glob:

  const metas       = import.meta.glob('/src/content/**/meta.json', { eager: true });
  const simulations = import.meta.glob('/src/content/**/Simulation.tsx');
  const logics       = import.meta.glob('/src/content/**/logic.mdx');
  const references   = import.meta.glob('/src/content/**/references.json', { eager: true });
  const traces       = import.meta.glob('/src/content/**/trace.json', { eager: true });
                        // OPTIONAL per concept — see Section 18. A concept
                        // with no matching trace.json simply gets `null`
                        // here; the registry must handle that as "no synced
                        // code view for this concept," not as an error.

Processing steps the agent must implement:
  1. For every meta.json match, parse the folder path into segments to derive
     section > parent chain > concept id.
  2. Build a tree data structure (ConceptNode[]) mirroring the folder
     hierarchy, sorted by each node's `order` field.
  3. Attach lazy-loading references (not the resolved component) for
     Simulation.tsx and logic.mdx — these stay as functions to be called by
     React.lazy() at route-render time, so each concept is its own code-split
     chunk.
  4. Attach the eagerly-loaded references.json content directly (small JSON,
     fine to bundle eagerly).
  4a. Attach the eagerly-loaded trace.json content directly where present
      (also small JSON, safe to bundle eagerly), plus a boolean
      `hasCodeTrace` flag on the ConceptNode so the UI can decide whether to
      render ConceptWorkbench.tsx (Section 18) or a plain Simulation.tsx.
  5. Export: 
       - `conceptTree: ConceptNode[]`      (for Sidebar)
       - `flatConceptIndex: ConceptNode[]`  (for search/command palette)
       - `getConceptByPath(path: string)`   (for the route resolver)
  6. This file must require ZERO manual edits when new content folders are
     added — it is purely derived from the filesystem scan.

4.8 Relationship to Section 18 (code trace) and Section 19 (Simulate the Logic)
  - trace.json (4.1) is the on-disk file; its full schema, the stepMap
    alignment mechanism, and the ConceptWorkbench/CodeTracePanel components
    that consume it are fully specified in Section 18 — not duplicated here
    to avoid drift between two copies of the same schema.
  - The "Simulate the Logic" Beta page (Section 19) does NOT use this plugin
    architecture at all — it has no content folder, no meta.json, and is
    generated live from user input. It is intentionally a separate system;
    do not try to route it through registry.ts.

4.7 Adding a new concept — the actual workflow the agent enables
  1. Create /src/content/<section>/<new-concept>/ with the 4 required files.
  2. Fill meta.json, write Simulation.tsx, write logic.mdx, add references.
  3. Save. Dev server hot-reloads; sidebar, routing, breadcrumbs, and search
     all pick it up automatically. Nothing else is touched.

5. ROUTING [LOCKED]
================================================================================
Using React Router v6, single dynamic catch-all pattern:

  /                                     -> Landing.tsx
  /workspace/*                          -> Workspace.tsx, which reads the
                                            wildcard remainder as the content
                                            path and resolves it via
                                            getConceptByPath()
  /simulate                             -> SimulateLogic.tsx (Beta, Section
                                            19). A separate top-level route,
                                            NOT nested under /workspace,
                                            since it has no content path to
                                            resolve and is a different mode
                                            of use (free-form, not tied to
                                            the concept library).

Inside Workspace.tsx:
  - Parse the URL path segments after /workspace/ into a content path
    (e.g. "algorithms/graph/shortest-path").
  - Look up the node via getConceptByPath().
  - If found: React.lazy()-load its Simulation.tsx and logic.mdx inside
    <Suspense> boundaries with skeleton/loading states.
  - If not found: render a clean 404 state inside the workspace shell (not a
    full page reload/redirect).
  - If path resolves to a section or intermediate folder with children but no
    own content: render an index/overview listing its children as cards.

Breadcrumb component derives its trail directly from the same path segments
— do not maintain a separate breadcrumb state.

6. THEMING & COLOR SYSTEM [LOCKED — FOLLOW EXACT VALUES]
================================================================================
6.1 Mechanism
- Theme state stored in React Context (features/theme/ThemeProvider.tsx) and
  mirrored to `localStorage` (key: "cs-sim-theme") purely for persistence
  across reloads — NOT tied to any user account.
- Active theme is applied via `data-theme="dark"` or `data-theme="light"`
  attribute on the `<html>` element.
- ALL colors are defined as CSS custom properties in /src/styles/tokens.css,
  scoped under `[data-theme="dark"]` and `[data-theme="light"]` selectors.
  Tailwind config maps utility classes to these variables (e.g.
  `bg-background`, `text-foreground`, `border-border`) — components must
  NEVER hardcode hex/rgb colors directly.

6.2 Semantic token layer (both themes must define all of these)
  --background
  --surface            (cards/panels — one step off background)
  --surface-hover
  --border
  --foreground          (primary text)
  --foreground-muted    (secondary/meta text)
  --accent-algorithms
  --accent-os
  --accent-networking
  --accent-systems
  --accent-languages
  --success
  --warning
  --error

6.3 Theme 1 — Dark ("exact Vercel" aesthetic)
  --background:        #0a0a0a
  --surface:            #111111
  --surface-hover:      #1a1a1a
  --border:             rgba(255,255,255,0.10)
  --foreground:         #ededed
  --foreground-muted:   #a1a1a1

  Contrast check (must hold):
    #ededed on #0a0a0a  -> ~18:1  (WCAG AAA, target: body text)
    #a1a1a1 on #0a0a0a  -> ~8:1   (AA/AAA, target: secondary/meta text)

6.4 Theme 2 — Light ("buttery" — NOT pure white, warm-toned)
  --background:        #faf8f3
  --surface:            #f0ece2
  --surface-hover:      #e8e2d4
  --border:             rgba(30,25,15,0.08)
  --foreground:         #1f1c17
  --foreground-muted:   #6b6558

  Contrast check (must hold):
    #1f1c17 on #faf8f3  -> ~15:1  (WCAG AAA, target: body text)
    #6b6558 on #faf8f3  -> ~6:1   (AA, target: secondary/meta text)

  Rule: never use pure #ffffff as background and never pure #000000 as text
  anywhere in the light theme — this is what breaks the "buttery" warmth.

6.5 Section accent colors — OKLCH, fixed hue across themes
Define these as OKLCH values (fall back to computed hex via a build step or
a CSS `color-mix`/`oklch()` function if targeting modern browsers only —
agent discretion on exact CSS implementation, but the perceptual values below
are fixed).

  Section        Hue anchor   Dark theme token value        Light theme token value
  Algorithms     Blue         oklch(70% 0.15 250)            oklch(55% 0.16 250)
  OS             Orange       oklch(72% 0.16 55)              oklch(58% 0.17 55)
  Networking     Green        oklch(72% 0.14 150)            oklch(55% 0.15 150)
  Systems        Purple       oklch(70% 0.15 300)            oklch(55% 0.16 300)
  Languages      Teal         oklch(72% 0.14 190)            oklch(55% 0.15 190)

  Rule (must be enforced in code review / component design): accent colors
  are used ONLY for icons, active/selected states, small badges, borders on
  hover, and highlight elements inside simulations (e.g. active graph node).
  Accent colors must NEVER be used as body/paragraph text color. Body text
  always uses --foreground / --foreground-muted regardless of section.

6.6 Typography tokens
  --font-sans: 'Inter', 'Geist', system-ui, sans-serif
  --font-mono: 'JetBrains Mono', 'IBM Plex Mono', monospace
  Use --font-mono for: pseudocode blocks, complexity notation (Big-O), code
  snippets inside logic.mdx, and any raw data/array visualizations in
  simulations (reinforces the "dev tool" feel).

6.7 Theme toggle UI
  Simple two-state icon toggle (sun/moon or similar) placed in the Workspace
  top bar / header. Instant switch, no page reload, animate the transition
  of background/surface colors over ~150-200ms for polish (avoid a harsh
  instant flash).

9. AI CHAT SYSTEM [LOCKED]
================================================================================
9.1 Serverless function — /api/chat.ts
- Single file, deployed as a Vercel/Netlify serverless function alongside
  the static SPA build (same repo, same deploy).
- Responsibilities:
    1. Receive POST request with { message: string, context: string,
       history: {role: string, content: string}[] }
    2. Read GEMINI_API_KEY from environment variables (server-side only,
       set in Vercel/Netlify project settings — never committed to repo,
       never exposed in any client bundle).
    3. Construct request to Google Gemini API, injecting `context` (the
       current concept's logic.mdx content + meta.json title/summary) into
       the system/instruction portion of the prompt so answers are grounded
       in what the user is currently viewing.
    4. Forward the model's response back to the client as JSON.
    5. Handle errors gracefully (rate limits, malformed input) and return
       clean error responses — never leak the API key or raw provider error
       internals to the client.
- CORS: restrict to same-origin (the deployed app's own domain) since this
  is not meant to be a public open API.

9.2 Client-side — features/chat/
- ChatPanel.tsx: renders message list + input box + send button, matches
  card-style compact layout described in Section 7.
- useChat.ts hook: manages local message state, calls lib/api.ts's
  `sendChatMessage()` which POSTs to `/api/chat`, handles loading/streaming
  state (streaming is [AGENT DISCRETION] — simple request/response is
  acceptable for v1; can upgrade to streamed tokens later without changing
  the architecture).
- Context injection: Workspace.tsx (or a shared context provider) tracks
  "currently active concept," and ChatPanel reads that to include the right
  context string with each message sent — this must update automatically as
  the user navigates concepts, without the user having to re-explain what
  they're looking at.
- No chat history is persisted beyond the browser session/tab lifetime (no
  auth, no backend storage — by design).

17. CONCEPT REQUEST FORM (GITHUB ISSUES INTEGRATION) [LOCKED]
================================================================================
17.1 Purpose
When a user searches or browses the whole app and cannot find a concept they
want, they can file a lightweight request instead of leaving empty-handed.
This is an intake mechanism, not an auto-publish pipeline (see 17.8).

17.2 Entry points (all open the same RequestConceptForm)
  - Footer link in the Sidebar: "Can't find a concept? Request one."
  - Empty-state of the Cmd+K search results ("No results — request this
    concept instead").
  - The Workspace 404 state, when a URL resolves to nothing in the registry.

17.3 Form fields (exact — do not add/remove without reason)
  - name              text, OPTIONAL (requester's name/handle; default to
                       "Anonymous" if left blank)
  - relatedSection     single-select, REQUIRED — options populated dynamically
                       from the registry's existing top-level sections
                       (algorithms / os / networking / systems / languages),
                       plus a literal "Other / New Section" option
  - conceptName        text, REQUIRED
  - logic              textarea, REQUIRED — free description of the concept/
                       algorithm/logic the requester wants simulated
  - references          repeatable field group, REQUIRED, min 1 / max 5
                       entries — each entry is { title: text, url: text }.
                       There is NO "verified" checkbox exposed to the
                       requester; verification stays an internal/manual step
                       done by whoever triages the resulting GitHub issue.

17.4 Client-side validation
  - All required fields must be non-empty before the submit button enables.
  - Each reference's `url` must pass a basic URL-format check.
  - At least 1 complete reference (title + valid url) required; up to 5.

17.5 Submission flow
  POST the form payload as JSON to /api/request-concept.ts, which:
    1. Re-validates the payload server-side (never trust client validation
       alone).
    2. Formats it into a GitHub Issue using a fixed markdown template, so
       every incoming request is consistent to triage:

         Title: [Concept Request] <conceptName>
         Body:
           **Requested by:** <name, or "Anonymous">
           **Section:** <relatedSection>
           **Logic:**
           <logic>

           **References:**
           - <title> — <url>
           - <title> — <url>

    3. Calls the GitHub REST API: POST /repos/{owner}/{repo}/issues
       - Applies labels: a fixed ["concept-request"] label plus a second
         label matching the section (e.g. "section:algorithms") so requests
         are filterable directly in GitHub's own issue UI without any extra
         tooling on your side.
    4. Returns a simple success/failure JSON response to the client.

17.6 Authentication (answers "do I need a repo/auth token?" — yes)
  - Required: a fine-grained GitHub Personal Access Token, scoped to ONLY
    this one repository, with ONLY the "Issues: write" permission — no
    broader repo access, no org-level access.
  - This token belongs to the project owner (you), not the end user. It is
    stored as an environment variable (`GITHUB_TOKEN`) in the serverless
    function's runtime settings on Vercel/Netlify — never committed to the
    repo, never sent to or readable from the client bundle.
  - End users never authenticate with GitHub in any way. This preserves the
    "no login" requirement everywhere else in the app — the token is a
    service credential for your backend function, not a user-facing login.

17.7 Anti-abuse measures (required, since this is a public write-capable
     endpoint)
  - Honeypot: include a hidden form field that real users never fill in and
    bots typically do; if it arrives populated, silently reject (200 OK,
    do nothing) rather than erroring, so bots don't learn to detect this.
  - Basic per-IP/per-session cooldown inside the serverless function (e.g.
    one accepted submission per 60 seconds) to avoid spam or exhausting
    GitHub's API rate limits.

17.8 What this flow is NOT
  This is intake-only. An accepted/triaged request still becomes a real
  concept through the normal manual process: a developer creates
  /src/content/<section>/<concept>/ with the required files per Section 4
  (and optionally trace.json per Section 18), same as authoring any other
  concept. The GitHub issue is the backlog/queue — nothing here auto-
  generates or auto-publishes content.

18. CODE EXECUTION TRACE SYSTEM (SIMULTANEOUS SIMULATION + CODE STEP-THROUGH)
    [LOCKED — EXISTING CONCEPTS ONLY, NOT AVAILABLE ON THE SIMULATE-THE-LOGIC
    PAGE]
================================================================================
18.1 Scope
This feature applies ONLY to pre-authored concepts living under
/src/content. It is explicitly NOT available on the Beta "Simulate the
Logic" page (Section 19) — that page has no real source code to trace,
only AI-generated abstract data.

18.2 What the user sees
Inside a concept's Simulation tab, when trace.json is present, two
synchronized panels appear instead of just one:
  - Primary panel: the existing hand-built abstract visual (Simulation.tsx)
    — bars moving, graph nodes lighting up, whatever the concept's own
    animation already does.
  - Secondary panel: the concept's actual C-style code (or exact runnable
    code, where the concept uses one) with the currently-executing line
    highlighted, plus a live variables table and call-stack list — the same
    mental model as stepping through a program in GDB inside VS Code.
  - ONE shared set of controls (Play / Pause / Step Forward / Step Back /
    Reset / Speed) drives BOTH panels at the same time — this is the
    "simultaneous" requirement from the original request.

18.3 Architecture — shared step state
  New component: /src/features/code-trace/ConceptWorkbench.tsx
  - Renders only when the current concept's registry entry has
    `hasCodeTrace: true` (see Section 4.6/4.8). Otherwise the app renders
    Simulation.tsx alone, unchanged from the original spec.
  - Owns a single `currentStep: number` piece of state, plus play/pause/
    speed state, and is the ONE place that renders <SimulationControls />
    for this concept when a trace exists (Simulation.tsx does not render
    its own controls in this mode — see the `externalStep` extension in
    Section 4.3).
  - Passes the correctly-mapped step index down to each child (see 18.4).

  New component: /src/features/code-trace/CodeTracePanel.tsx
  - Renders trace.json's `sourceCode` with syntax highlighting, using the
    app's shared --font-mono token for visual consistency with the rest of
    the app's code blocks (pseudocode in logic.mdx, etc.).
  - Highlights the line for the current trace step.
  - Renders a variables table and call-stack list sourced directly from
    that step's data in trace.json — no computation happens at runtime,
    it's pure lookup/render.

18.4 Step alignment — stepMap
  An abstract visual "step" (e.g. "highlight the smaller of two subarrays")
  rarely maps 1:1 to a single line of code execution. Each trace.json
  therefore includes a `stepMap` array: `stepMap[abstractStepIndex]` gives
  the corresponding index into trace.json's `steps` array (or a [start,end]
  range if one abstract step spans several execution steps). 
  ConceptWorkbench uses `stepMap` to translate the one shared `currentStep`
  into: the abstract index (passed to Simulation.tsx via `externalStep`)
  and the mapped trace index (passed to CodeTracePanel).

18.5 trace.json schema (optional 5th content file — full schema, see also
     the pointer in Section 4.1/4.8)
  Location: /src/content/<section>/.../<concept>/trace.json

  {
    "language": "c",
    "sourceCode": "void mergeSort(int array[], int start, int end) {\n  ...\n}",
    "steps": [
      {
        "line": 4,
        "variables": { "start": 0, "end": 7, "middle": 3 },
        "callStack": ["mergeSort(0,7)", "mergeSort(0,3)"],
        "stdout": ""
      }
    ],
    "stepMap": [0, 0, 1, 2, 2, 3]
  }

  Notes:
  - `line` is 1-indexed against `sourceCode`.
  - `variables` is a flat key/value object — keep it flat per step; nested
    structures (arrays, pointers) can be represented as plain JSON
    values/strings, CodeTracePanel is a dumb renderer, not an interpreter.
  - Presence of this file alone is what flips the UI into split view; no
    other configuration is needed anywhere else in the app.

18.6 How trace.json actually gets generated — OFFLINE / DEV-TIME ONLY, never
     a production service
  Because the concept library is a fixed, pre-authored set (never arbitrary
  user-submitted code), there is no need for live code execution in
  production, ever. trace.json is generated once, by the content author, on
  their own machine, and committed to the repo as a plain static file —
  same as meta.json or references.json.

  Recommended tool: OnlinePythonTutor
  (github.com/pgbovine/OnlinePythonTutor, GPLv3), used strictly as a LOCAL
  authoring aid, never deployed as a production service. It already
  produces exactly this kind of step-by-step execution trace for Python,
  Java, JavaScript, TypeScript, Ruby, C, and C++.

  Correction/clarification on its C/C++ backend specifically: it does NOT
  use gdb — it runs on a hacked build of Valgrind, packaged as a Docker
  service the project calls "cokapi" (reference build:
  github.com/JuezUN/opt-cpp-backend). Implications:
    - Requires a container runtime to run locally — see 18.7 for the
      Fedora-specific setup this team needs.
    - Licensed GPLv3. Running it locally as a one-off authoring tool and
      committing only its JSON *output* (not its source) to this project
      does not place this project under GPL obligations — you're using its
      output as data, not distributing or linking against the tool itself.
      If the team ever considers running it as a live, always-on backend
      service instead of a local authoring tool, revisit this at that time
      (GPLv3 lacks AGPL's network-use clause, but it's still worth a second
      look before making that change).
    - Not serverless-friendly (needs a persistent process/container), which
      is exactly why it stays out of the production architecture entirely —
      see Section 2.1's function list: only /api/chat.ts,
      /api/request-concept.ts, and the Simulate the Logic worker exist in
      production. Nothing related to code tracing runs in production.

  Alternative, often faster in practice: for concepts whose "code" is
  closely mirrored by the abstract simulation's own internal state machine
  (common when displaying C-style pseudocode rather than exact compiled C),
  it's often quicker for the content author to hand-write the `steps` array
  directly to match the animation beat-by-beat, rather than round-tripping
  through Valgrind/OnlinePythonTutor at all. [AGENT DISCRETION] per concept,
  on which approach is faster for that specific piece of content.

18.7 Fedora-specific local setup for content authors
  Both current team members develop on Fedora Linux. Fedora ships Podman by
  default, not Docker Engine, which matters specifically for running
  OnlinePythonTutor's C/C++ Docker-based backend locally as a one-off
  authoring tool:
    - Simplest path: `sudo dnf install podman-docker` — this provides a real
      `docker` CLI shim backed by Podman, so the project's existing
      `docker build` / `docker run` instructions from its README work
      completely unmodified, no translation needed.
    - Alternative without the shim: substitute `podman` for `docker`
      directly in single-container commands (e.g.
      `podman build -t opt-cpp-backend .` then
      `podman run -p 4000:4000 opt-cpp-backend`), since Podman is
      intentionally Docker-CLI-compatible for the common cases this tool
      needs.
    - Fedora's default rootless Podman mode is sufficient here — this is a
      local, one-off, non-privileged authoring task, no need to run rootful.
    - If a contributor genuinely wants Docker Engine itself on Fedora
      instead of Podman, it is NOT in Fedora's default repos and must be
      installed from Docker's official Fedora repo — do not `dnf install
      docker`, that pulls an unrelated package by that name on some Fedora
      releases.
    - This entire setup only matters for the OFFLINE trace-authoring
      workflow. It has zero bearing on the production deploy, which has no
      Docker/Podman dependency whatsoever.

18.8 Summary of what actually ships to production for this feature
  - trace.json files (static, one per concept that has this feature),
    served exactly like references.json — effectively zero runtime cost.
  - ConceptWorkbench.tsx + CodeTracePanel.tsx — pure client-side rendering,
    no backend calls of any kind.
  - No new serverless function, no new environment variable, no container
    runtime anywhere in production. This entire feature is front-end
    rendering plus static data once a concept's trace has been authored.

19.5 Critical design constraint — never let the model generate arbitrary
     rendering code or execute arbitrary user code
  Constrain the LLM's output to a small, fixed set of JSON shapes the app
  already knows how to render (reusing the same visual primitives built for
  Section 10's simulation engine mapping):

  {
    "type": "array-sequence" | "graph-sequence" | "stack-sequence" | "queue-sequence",
    "steps": [ /* shape depends on "type", always plain data — NEVER
                  executable code, HTML, or markup of any kind */ ]
  }

  The system prompt sent to the model must explicitly restrict it to these
  shapes only. The Worker (or client, whichever validates first) must check
  the returned JSON against a schema (e.g. zod) BEFORE any rendering is
  attempted. On any malformed or out-of-schema response: show a friendly
  "Couldn't simulate that — try rephrasing" message. Never attempt to
  render an unrecognized shape, and never `eval`/`dangerouslySetInnerHTML`
  anything derived from the model's output.

19.6 Rendering
  Reuse the existing per-type visual building blocks already required for
  the concept library (the same "array bars," "graph nodes," "stack/queue
  blocks" primitives referenced in Section 10's engine mapping) — driven by
  the AI-generated `steps` array instead of a hand-authored one. This means
  Section 19 adds very little new rendering code; it's mostly a thin AI
  layer plus schema validation on top of infrastructure the concept library
  already needs to exist.

19.7 No connection to Section 18
  This page never has real source code to trace — the user only described
  logic in prose, nothing was compiled or executed. The ConceptWorkbench /
  CodeTracePanel split view from Section 18 does NOT apply here. Simulate
  the Logic only ever shows the single generic abstract visual, no code
  panel, ever.

19.8 No persistence
  Each simulation is generated fresh per request; nothing is saved, no
  history across page loads or sessions — consistent with the whole app's
  no-auth, no-database stance.

21. PHASE 3 / FUTURE CONSIDERATION — LIVE CODE EXECUTION IN PRODUCTION
    [NOT IN SCOPE FOR THE CURRENT BUILD — DOCUMENTED FOR FUTURE REFERENCE ONLY]
================================================================================
21.1 Status
This section exists purely to record a deliberate tradeoff decision so it
isn't relitigated from scratch later. Nothing in this section should be
built as part of the current spec (Sections 1–20). It only becomes relevant
if the team later decides live code execution needs to run in production
rather than being precomputed offline (Section 18.6).

The moment this stops being a "future consideration" and becomes real scope,
this section should be rewritten as [LOCKED] with the same level of detail
as Sections 17–19, not left as a loose sketch.

21.2 The decision hinges on WHO submits the code — two different problems
  Before building anything here, classify which scenario actually applies,
  since they have almost nothing in common architecturally:

  SCENARIO A — Trace generation stays limited to trusted content, just
  triggered through the app instead of a developer's local machine
    Example: an internal /admin tool where a content author pastes a
    concept's code and the app generates trace.json for them, instead of
    running OnlinePythonTutor/Docker locally per Section 18.7.
    - Input is still only ever code the team itself wrote — same trust
      level as the current offline workflow, just a UX convenience.
    - Host the cokapi/Valgrind service as a small persistent container on
      a platform that scales to zero when idle — Google Cloud Run, Fly.io,
      or Railway are reasonable fits, keeping cost near-zero between
      authoring sessions.
    - Gate the trigger endpoint behind a shared secret header or IP
      allowlist — full user auth is NOT required here, since this is an
      internal tool, not a public-facing feature.
    - Sandboxing needs are modest: basic CPU/memory/timeout limits are
      sufficient, since the input is already trusted. Real adversarial
      sandboxing (21.3) is not required for this scenario.
    - This is the lower-risk, lower-effort of the two paths and could
      reasonably be picked up as a Phase 3 item on its own, independent of
      Scenario B.

  SCENARIO B — End users submit or edit code that is then actually executed
    Example: allowing a user to edit a concept's code inline and re-run it,
    or wiring real execution into the Simulate the Logic page (Section 19)
    instead of constrained AI-generated abstract data.
    - Input is untrusted. This is a genuinely hard security problem, not a
      feature to bolt on casually — treat it with the same seriousness as
      any other user-facing execution surface.
    - Serverless functions (Vercel/Netlify, as used everywhere else in this
      app) are NOT suitable — they can't reliably run persistent processes
      or spawn compilers/Valgrind, and their execution-time limits work
      against this use case.
    - Do NOT adapt OnlinePythonTutor's cokapi/Valgrind setup for this.
      It was built for trusted educational/classroom use, not hardened for
      public multi-tenant execution.
    - Prefer purpose-built sandboxed execution engines that have already
      solved the isolation problem: Judge0 or Piston (both open source,
      both use OS-level isolation such as the `isolate` sandbox with
      seccomp+cgroups). Note the tradeoff: both give final stdout, not a
      step-by-step variable trace — matching Section 18's GDB-style trace
      quality here would mean extending one of them, not using it as-is.
    - For genuinely strong isolation, look at gVisor or Firecracker
      microVMs (the technology underlying AWS Lambda and Fly.io's own
      infrastructure) — each execution request gets a throwaway, network-
      isolated, resource-capped sandbox destroyed immediately after use.

21.3 Non-negotiable requirements IF Scenario B is ever built
  - Strict per-request CPU, memory, and wall-clock time limits.
  - No network egress from inside the execution sandbox.
  - No persistent filesystem — sandbox is destroyed after each execution.
  - Rate limiting per IP/session is the ONLY abuse lever available, since
    this app has no user accounts (Section 20) — public code-execution
    endpoints are a known target for cryptomining and denial-of-service
    attempts, and this app has no login wall to fall back on.
  - Reference point: Judge0 in its default configuration has had real
    sandbox-escape vulnerabilities reported historically — this is cited
    as a reminder that "just run it in a container" is not sufficient on
    its own; proper isolation (seccomp/gVisor/Firecracker-class tooling)
    is required, not optional hardening.

21.4 Cost/infrastructure implications
  Scenario B is a real departure from this app's near-zero-cost static-
  hosting model (Section 2/Section 14). Budget and infrastructure ownership
  should be explicitly revisited before starting this work — it is not a
  drop-in addition to the existing Vercel/Netlify + Cloudflare Workers
  setup.

21.5 Recommendation
  Do not build either scenario as part of the current spec. If/when
  revisited: Scenario A is a reasonable, low-risk Phase 3 pickup on its own
  (content-authoring convenience only). Scenario B should only be pursued
  if there's a specific, validated product reason for users to run their
  own code somewhere in the app — it is a meaningfully larger feature than
  anything else currently in this document, both in engineering effort and
  in ongoing security responsibility.
