1. PROJECT OVERVIEW
================================================================================
Name: CS Concept Simulator (working title — agent may propose alternatives)

What it is: A web application for learning computer science concepts through
interactive visual simulations. Organized into top-level sections (Algorithms,
Operating Systems, Networking, System Programming, Languages, etc.), each
containing a hierarchy of concepts and sub-concepts, similar to a file/directory
tree. Each concept page contains: an interactive simulation, a logic/explanation
write-up, and a curated references list. A persistent AI chat side-panel is
available throughout the working area to help the user understand the concept
they are currently viewing.

Core requirements:
- NO login / NO authentication anywhere. App opens directly into content.
- Landing page with a 3D animated hero, then a direct entry into the working
  "Workspace" area (IDE-like 3-pane layout).
- Must be trivially extensible: adding a brand-new concept/simulation should
  require ONLY adding new content files — no changes to routing code, sidebar
  code, or app shell code.
- Visual identity: sharp, professional, "native OS app" feel — NOT a generic
  marketing website. Reference aesthetic: Vercel Dashboard / Linear / VS Code.
- Two themes: exact Vercel-style dark theme, and a warm "buttery" light theme.
  Full color system detailed in Section 6.

================================================================================

  SVG, Canvas (via Konva.js if needed), or D3.js underneath as required by the

- Content authoring format: MDX (for logic write-ups that may embed live
  React components inline).
- AI chat backend: ONE serverless function (see Section 9). Provider: Google
  Gemini API. The API key lives ONLY in the serverless function's environment
  variables — it must NEVER be shipped to or callable from client-side code.
- Icons: Lucide React.
- Fonts: Inter or Geist for UI text; JetBrains Mono or IBM Plex Mono for code/
  pseudocode blocks.
- Command palette: cmdk library (pairs natively with shadcn).
- Deployment target: Vercel or Netlify (both support static SPA hosting +
  a single serverless function in the same deploy — no separate backend
  service required).

Explicitly OUT of scope / do not add:
- No Redux / Zustand / MobX — state needs are minimal (see Section 11).
- No database. No user accounts. No persistent server-side storage.
- No SSR/SSG — this is a pure client-rendered SPA.

4. CONTENT PLUGIN ARCHITECTURE (CORE MECHANISM) [LOCKED]
================================================================================
This is the most important architectural pattern in the app. It is what
guarantees "adding a new concept = only add files, no code changes."

4.1 Folder convention
Every concept (and sub-concept) is a folder under /src/content/<section>/
containing exactly these 4 files:
  - meta.json          (required)
  - Simulation.tsx      (required)
  - logic.mdx           (required)
  - references.json     (required, may be an empty array initially)

Nesting is allowed arbitrarily deep to represent hierarchy, e.g.:
  /content/algorithms/graph/shortest-path/dijkstra/...
The registry builder must support arbitrary depth, not just 2 levels.

4.2 meta.json schema
{
  "id": "shortest-path",                 // must match folder name, kebab-case
  "title": "Shortest Path Algorithms",
  "section": "algorithms",               // top-level section key
  "parentPath": "algorithms/graph",      // full path minus this folder, for
                                          // breadcrumb + sidebar nesting
  "order": 3,                            // sort order within its parent
  "difficulty": "intermediate",          // "beginner" | "intermediate" | "advanced"
  "tags": ["graph", "greedy", "dijkstra", "bellman-ford"],
  "prerequisites": ["algorithms/graph/bfs-dfs"],  // array of concept ids/paths
  "summary": "One or two sentence description shown in sidebar hover / search results",
  "accentSection": "algorithms"          // used to pick the correct accent
                                          // color token (see Section 6.4)
}

4.3 Simulation.tsx contract
- Must be a default-exported React component with NO required props (it may
  accept optional props for things like initial state, but must render fully
  standalone with defaults).
- Must be self-contained: all state, animation logic, and controls (play/
  pause/step/reset/speed slider) live inside this file or its own local
  sub-components in the same folder if it grows large (e.g. create a
  /shortest-path/components/ subfolder for internal pieces — this is allowed
  and does not break the plugin contract, since the loader only ever imports
  Simulation.tsx as the entry point).
- Must use Framer Motion for transitions to keep visual language consistent
  app-wide, and must consume the theme's CSS variables for colors (never
  hardcode hex values inside a simulation file).
- Must implement standard simulation controls as a shared UI convention:
  Play / Pause / Step Forward / Step Back / Reset / Speed control. Build a
  reusable <SimulationControls /> component in /components/ui and have every
  Simulation.tsx use it, so all simulations feel consistent.

4.4 logic.mdx contract
- Contains the written explanation: problem definition, step-by-step logic,
  pseudocode (in fenced code blocks using the mono font), time/space
  complexity, common pitfalls.
- Since it's MDX, it MAY embed small inline React components (e.g. a mini
  complexity comparison table component) — but the main interactive
  simulation always lives in Simulation.tsx, not inline in the MDX.

4.5 references.json schema
[
  {
    "title": "Dijkstra's Algorithm — CLRS Chapter 24",
    "type": "book",                 // "paper" | "video" | "doc" | "tool" | "book" | "article"
    "source": "Introduction to Algorithms (CLRS)",
    "url": "https://...",
    "verified": true                // manually curated/verified flag, shown as a badge
  }
]

4.6 registry.ts — auto-build logic
Implemented in /src/lib/contentLoader.ts using Vite's import.meta.glob:

  const metas       = import.meta.glob('/src/content/**/meta.json', { eager: true });
  const simulations = import.meta.glob('/src/content/**/Simulation.tsx');
  const logics       = import.meta.glob('/src/content/**/logic.mdx');
  const references   = import.meta.glob('/src/content/**/references.json', { eager: true });

Processing steps the agent must implement:
  1. For every meta.json match, parse the folder path into segments to derive
     section > parent chain > concept id.
  2. Build a tree data structure (ConceptNode[]) mirroring the folder
     hierarchy, sorted by each node's `order` field.
  3. Attach lazy-loading references (not the resolved component) for
     Simulation.tsx and logic.mdx — these stay as functions to be called by
     React.lazy() at route-render time, so each concept is its own code-split
     chunk.
  4. Attach the eagerly-loaded references.json content directly (small JSON,
     fine to bundle eagerly).
  5. Export: 
       - `conceptTree: ConceptNode[]`      (for Sidebar)
       - `flatConceptIndex: ConceptNode[]`  (for search/command palette)
       - `getConceptByPath(path: string)`   (for the route resolver)
  6. This file must require ZERO manual edits when new content folders are
     added — it is purely derived from the filesystem scan.

4.7 Adding a new concept — the actual workflow the agent enables
  1. Create /src/content/<section>/<new-concept>/ with the 4 required files.
  2. Fill meta.json, write Simulation.tsx, write logic.mdx, add references.
  3. Save. Dev server hot-reloads; sidebar, routing, breadcrumbs, and search
     all pick it up automatically. Nothing else is touched.

5. ROUTING [LOCKED]
================================================================================
Using React Router v6, single dynamic catch-all pattern:

  /                                     -> Landing.tsx
  /workspace/*                          -> Workspace.tsx, which reads the
                                            wildcard remainder as the content
                                            path and resolves it via
                                            getConceptByPath()

Inside Workspace.tsx:
  - Parse the URL path segments after /workspace/ into a content path
    (e.g. "algorithms/graph/shortest-path").
  - Look up the node via getConceptByPath().
  - If found: React.lazy()-load its Simulation.tsx and logic.mdx inside
    <Suspense> boundaries with skeleton/loading states.
  - If not found: render a clean 404 state inside the workspace shell (not a
    full page reload/redirect).
  - If path resolves to a section or intermediate folder with children but no
    own content: render an index/overview listing its children as cards.

Breadcrumb component derives its trail directly from the same path segments
— do not maintain a separate breadcrumb state.

6. THEMING & COLOR SYSTEM [LOCKED — FOLLOW EXACT VALUES]
================================================================================
6.1 Mechanism
- Theme state stored in React Context (features/theme/ThemeProvider.tsx) and
  mirrored to `localStorage` (key: "cs-sim-theme") purely for persistence
  across reloads — NOT tied to any user account.
- Active theme is applied via `data-theme="dark"` or `data-theme="light"`
  attribute on the `<html>` element.
- ALL colors are defined as CSS custom properties in /src/styles/tokens.css,
  scoped under `[data-theme="dark"]` and `[data-theme="light"]` selectors.
  Tailwind config maps utility classes to these variables (e.g.
  `bg-background`, `text-foreground`, `border-border`) — components must
  NEVER hardcode hex/rgb colors directly.

6.2 Semantic token layer (both themes must define all of these)
  --background
  --surface            (cards/panels — one step off background)
  --surface-hover
  --border
  --foreground          (primary text)
  --foreground-muted    (secondary/meta text)
  --accent-algorithms
  --accent-os
  --accent-networking
  --accent-systems
  --accent-languages
  --success
  --warning
  --error

6.3 Theme 1 — Dark ("exact Vercel" aesthetic)
  --background:        #0a0a0a
  --surface:            #111111
  --surface-hover:      #1a1a1a
  --border:             rgba(255,255,255,0.10)
  --foreground:         #ededed
  --foreground-muted:   #a1a1a1

  Contrast check (must hold):
    #ededed on #0a0a0a  -> ~18:1  (WCAG AAA, target: body text)
    #a1a1a1 on #0a0a0a  -> ~8:1   (AA/AAA, target: secondary/meta text)

6.4 Theme 2 — Light ("buttery" — NOT pure white, warm-toned)
  --background:        #faf8f3
  --surface:            #f0ece2
  --surface-hover:      #e8e2d4
  --border:             rgba(30,25,15,0.08)
  --foreground:         #1f1c17
  --foreground-muted:   #6b6558

  Contrast check (must hold):
    #1f1c17 on #faf8f3  -> ~15:1  (WCAG AAA, target: body text)
    #6b6558 on #faf8f3  -> ~6:1   (AA, target: secondary/meta text)

  Rule: never use pure #ffffff as background and never pure #000000 as text
  anywhere in the light theme — this is what breaks the "buttery" warmth.

6.5 Section accent colors — OKLCH, fixed hue across themes
Define these as OKLCH values (fall back to computed hex via a build step or
a CSS `color-mix`/`oklch()` function if targeting modern browsers only —
agent discretion on exact CSS implementation, but the perceptual values below
are fixed).

  Section        Hue anchor   Dark theme token value        Light theme token value
  Algorithms     Blue         oklch(70% 0.15 250)            oklch(55% 0.16 250)
  OS             Orange       oklch(72% 0.16 55)              oklch(58% 0.17 55)
  Networking     Green        oklch(72% 0.14 150)            oklch(55% 0.15 150)
  Systems        Purple       oklch(70% 0.15 300)            oklch(55% 0.16 300)
  Languages      Teal         oklch(72% 0.14 190)            oklch(55% 0.15 190)

  Rule (must be enforced in code review / component design): accent colors
  are used ONLY for icons, active/selected states, small badges, borders on
  hover, and highlight elements inside simulations (e.g. active graph node).
  Accent colors must NEVER be used as body/paragraph text color. Body text
  always uses --foreground / --foreground-muted regardless of section.

6.6 Typography tokens
  --font-sans: 'Inter', 'Geist', system-ui, sans-serif
  --font-mono: 'JetBrains Mono', 'IBM Plex Mono', monospace
  Use --font-mono for: pseudocode blocks, complexity notation (Big-O), code
  snippets inside logic.mdx, and any raw data/array visualizations in
  simulations (reinforces the "dev tool" feel).

6.7 Theme toggle UI
  Simple two-state icon toggle (sun/moon or similar) placed in the Workspace
  top bar / header. Instant switch, no page reload, animate the transition
  of background/surface colors over ~150-200ms for polish (avoid a harsh
  instant flash).

9. AI CHAT SYSTEM [LOCKED]
================================================================================
9.1 Serverless function — /api/chat.ts
- Single file, deployed as a Vercel/Netlify serverless function alongside
  the static SPA build (same repo, same deploy).
- Responsibilities:
    1. Receive POST request with { message: string, context: string,
       history: {role: string, content: string}[] }
    2. Read GEMINI_API_KEY from environment variables (server-side only,
       set in Vercel/Netlify project settings — never committed to repo,
       never exposed in any client bundle).
    3. Construct request to Google Gemini API, injecting `context` (the
       current concept's logic.mdx content + meta.json title/summary) into
       the system/instruction portion of the prompt so answers are grounded
       in what the user is currently viewing.
    4. Forward the model's response back to the client as JSON.
    5. Handle errors gracefully (rate limits, malformed input) and return
       clean error responses — never leak the API key or raw provider error
       internals to the client.
- CORS: restrict to same-origin (the deployed app's own domain) since this
  is not meant to be a public open API.

9.2 Client-side — features/chat/
- ChatPanel.tsx: renders message list + input box + send button, matches
  card-style compact layout described in Section 7.
- useChat.ts hook: manages local message state, calls lib/api.ts's
  `sendChatMessage()` which POSTs to `/api/chat`, handles loading/streaming
  state (streaming is [AGENT DISCRETION] — simple request/response is
  acceptable for v1; can upgrade to streamed tokens later without changing
  the architecture).
- Context injection: Workspace.tsx (or a shared context provider) tracks
  "currently active concept," and ChatPanel reads that to include the right
  context string with each message sent — this must update automatically as
  the user navigates concepts, without the user having to re-explain what
  they're looking at.
- No chat history is persisted beyond the browser session/tab lifetime (no
  auth, no backend storage — by design).

10. SIMULATION ENGINE MAPPING [AGENT DISCRETION WITHIN THIS TABLE]
================================================================================
Use this table as the default choice per concept type. Framer Motion is used
for transitions/animation timing in all cases; the table below is about the
underlying rendering primitive.

  Concept type                              Recommended engine
  ------------------------------------------------------------------
  Graph/tree algorithms                     React Flow, or D3.js
  (Dijkstra, BFS/DFS, MST, A*)               force-directed graph

  Sorting / array-based visualizations       SVG + CSS/Framer Motion
  (merge sort, quicksort, heap sort)         transitions on bar elements

  OS concepts                                 Canvas (via Konva.js wrapper)
  (scheduling, paging, memory allocation)     for block/timeline diagrams

  Networking                                  SVG + Framer Motion, animated
  (packet flow, TCP handshake, routing)       paths/markers along SVG paths

  Low-level/systems programming               Custom SVG or Canvas byte-grid
  (memory layout, pointers, stack/heap)       visualizations

  Language-specific concept demos             Depends on concept — default to
  (closures, async/event loop, etc.)          SVG/Framer Motion unless a
                                               specific need dictates Canvas

All simulations must consume theme CSS variables for every color used
(no hardcoded hex inside any Simulation.tsx) and must implement the shared
<SimulationControls /> (Play/Pause/Step/Reset/Speed) described in 4.3.

11. STATE MANAGEMENT RULES [LOCKED]
================================================================================
- No global state library (Redux/Zustand/MobX). This app's state needs are
  small enough for React Context + URL state.
- Theme (dark/light): React Context, mirrored to localStorage for
  persistence only (not account-tied).

13. ACCESSIBILITY & READABILITY RULES [LOCKED]
================================================================================
- All text/background pairs must meet at least WCAG AA (4.5:1 for body text,
  3:1 for large text/UI elements) — the values specified in Section 6 already
  satisfy this; do not substitute colors that weaken these ratios.
- Never rely on accent color alone to convey state (e.g. "active" sidebar
  item) — always pair with a secondary cue (bold weight, left border, icon
  change) for colorblind users.
- All interactive simulation controls must be keyboard-operable (Play/Pause/
  Step/Reset reachable via Tab + Enter/Space, not mouse-only).
- Respect `prefers-reduced-motion`: simulations and page transitions should
  offer a reduced-motion fallback (instant state changes instead of animated
  transitions) where feasible.

17. NON-GOALS / EXPLICIT CONSTRAINTS (DO NOT IMPLEMENT)
================================================================================
- No user authentication, accounts, or login flow of any kind.
- No server-side database or persistent storage beyond the browser session.
- No CMS or admin panel for managing content — content is managed purely by
  adding/editing files in /src/content, by developers, via git.
- No SSR/SSG — pure client-rendered SPA.
- No second backend service beyond the single /api/chat.ts serverless
  function.
- Do not hardcode any color value outside of /src/styles/tokens.css.
- Do not hardcode sidebar/navigation structure anywhere — it must always be
  derived from the content registry.
